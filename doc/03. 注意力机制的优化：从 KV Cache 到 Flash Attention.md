# 一、KV Cache

## 1.1 传统注意力的显存与算力瓶颈

解码器的自回归生成是一个逐 Token 的过程：给定已有输入，模型预测下一个 Token，再将其拼接到输入中进行下一轮预测。

![alt text](assets/自回归生成.gif)

这个过程是**因果的**——每个 Token 只能关注自己及之前的 Token（这也是自然语言的时序特性）。

![alt text](<assets/Pasted image 20260318221928.png>)

问题在于：这种自回归方式会**重复大量已完成的计算**。我们可以放大掩码缩放点积注意力的计算过程来直观理解：

![](../../80_System/Attachments/Accessories/自回归生成放大.gif)

可以看到，每一步生成时我们都在重新计算之前所有 Token 的 K 和 V，但实际上**只有新 Token 的注意力是真正需要计算的**。

此外，随着上下文长度 $L$ 的增加（如从 4 K 增加到 128 K），注意力矩阵的显存占用呈 $O(L^2)$ 增长。传统 Attention 需要将完整的注意力矩阵写入 GPU HBM（高带宽内存），再读回进行 Softmax 计算，频繁的读写操作既耗时又极易导致 OOM。

## 1.2 KV Cache：以空间换时间

**KV Cache（键值缓存）** 的核心思想是：缓存已计算过的 K、V，避免每步都从头算起。

运行机制分两个阶段：

1. **Prefill（首次填充）**：将 Prompt 中所有 Token 的 K、V 一次性计算完毕，存入显存缓存。
2. **Decode（逐步解码）**：生成第 $n$ 个 Token 时，**仅计算当前 Token** 的 $q_n, k_n, v_n$，将 $k_n, v_n$ 追加到缓存末尾，注意力变为 $q_n$ 与完整 $[K_{cache}; k_n]$ 的交互。

![alt text](<assets/KV Cache.gif>)

> 上图中，🟣紫色是从缓存中获取的，🟢绿色是当前步新计算的，⬜灰色是被因果掩码屏蔽的（无需计算）。可以清晰地看到 KV Cache 消除了大量重复计算。

## 1.3 为什么不缓存 Q？

在注意力机制中，Q 代表**当前 Token 的查询意图**，它在每一步都是全新的：

- 生成第 10 个词时，Q₁₀ 查询的是"第 10 个词与前 9 个词的关系"；
- 生成第 11 个词时，Q₁₁ 查询的是"第 11 个词与前 10 个词的关系"。

每一步的 Q 都不同，也不会被后续步骤复用（因果性），因此**缓存 Q 没有意义**。

## 1.4 KV Cache 的显存挑战

KV Cache 虽然加速了推理，但它本身也会占用大量显存。随着 Batch Size 增大（提高吞吐）和序列长度增长（处理长文档），KV Cache 的体积往往**超过模型权重本身**，成为限制并发量的主要瓶颈。

这直接催生了对注意力架构的优化——如何在保持质量的前提下，减小 KV Cache 的体积？

# 二、MHA、MQA 和 GQA：KV 头数的权衡

![alt text](<assets/Pasted image 20260318222956.png>)

## 2.1 Multi-Head Attention (MHA)

- 假设有 $H$ 个注意力头，则有 $H$ 组独立的 Q、K、V 投影。
- **优势**：每个 Q 头都有专属的 KV 空间，能独立捕捉不同的语义特征（语法结构、指代关系等），表达能力最强。
- **缺点**：推理时需要缓存 $H$ 组 KV，显存占用极大，严重限制最大 Batch Size。

## 2.2 Multi-Query Attention (MQA)

- 保留 $H$ 个 Q 头，但**所有 Q 头共享同一个** K 头和 V 头。
- KV Cache 直接缩小 $H$ 倍（如 32 头模型，KV Cache 仅为 MHA 的 1/32）。
- **缺点**：所有 Q 头被迫在同一个 KV 子空间中检索，表达能力受限，可能导致生成质量下降，且训练初期容易不收敛。

## 2.3 Grouped-Query Attention (GQA)：折中方案

- 将 $H$ 个 Q 头分成 $G$ 个组，每组 $H/G$ 个 Q 头**共享一组** K 和 V。
- **优势**：
    1. **精度接近 MHA**：保留多组 KV 头（如 8 组），模型仍能捕捉多样语义，性能损失微乎其微。
    2. **速度接近 MQA**：KV Cache 降至 MHA 的 $1/G$（如 $G=4$ 则为 1/4），大幅降低显存带宽需求。
    3. **平滑升级**：可通过 Up-training 将现有 MHA 模型低成本转换为 GQA 模型。

> [!tip] 计算量并未减少
> 在实际计算注意力时，MQA/GQA 会通过 `repeat` / `broadcast` 将 KV 头扩展回与 Q 相同的头数，因此 $Q \cdot K^T$ 和 $\text{attn} \cdot V$ 的 FLOPs 三者几乎一样。GQA 的收益主要体现在**显存占用和内存带宽**上，而非计算量。

# 三、Flash Attention：IO 层面的优化

Flash Attention 是一种**工程优化的注意力实现**，数学本质与标准注意力完全相同，但在 GPU 内存访问模式上做了根本性改进：

| 维度 | 标准注意力 | Flash Attention |
|------|-----------|-----------------|
| **中间矩阵** | 将 $L \times L$ 注意力矩阵写入 HBM | 分块（Tiling）计算，**不**将完整矩阵写入 HBM |
| **显存** | $O(L^2)$ | $O(L)$（仅需块级缓冲区） |
| **IO 次数** | 多次读写 HBM | 尽量在 SRAM 中完成计算，减少 HBM 访问 |

核心思想：将 Q、K、V 分成小块，在 GPU 的 SRAM（片上高速缓存）中完成 softmax 和加权求和，通过 **online softmax** 技巧实现流式计算，避免显存瓶颈。

> [!note] 使用条件
> Flash Attention 最适合**训练阶段**或 Prefill 阶段（长序列并行计算）。在逐 Token 解码时（seq_len=1），没有足够的数据来发挥分块优势，通常退回标准注意力。

# 四、显存占用计算

## 4.1 核心参数定义

| **符号**     | **含义**                 | **单位/说明**                    |
| ------------ | ------------------------ | -------------------------------- |
| $P_{model}$  | 模型参数量               | Billions ( $10^9$ )                |
| $L_{seq}$    | 上下文序列长度           | Tokens                           |
| $B$          | 批处理大小 (Batch Size)  | 整数                             |
| $D_{model}$  | 隐藏层维度 (Hidden Size) | 整数                             |
| $N_{layers}$ | 模型层数                 | 整数                             |
| $N_{heads}$  | Query 注意力头数          | 整数                             |
| $N_{kv}$     | KV 注意力头数             | MHA 时 $= N_{heads}$，GQA 时 $< N_{heads}$ |
| $D_{head}$   | 单头维度                 | 通常 $D_{model} / N_{heads}$     |
| $S_{prec}$   | 数据精度大小             | Bytes（FP 16=2，FP 32=4，INT 8=1）   |

## 4.2 显存占用公式

总推理显存 $M_{total}$ 由三部分组成：

$$M_{total} = M_{weights} + M_{kv\_cache} + M_{activation}$$

> 推理阶段 $M_{activation}$ 相对较小，主要由权重和 KV Cache 主导；在长序列场景下 KV Cache 往往成为最大头。

**模型权重显存：**

$$M_{weights} \approx P_{model} \times S_{prec}$$

**KV Cache 显存（体现 GQA 的优势）：**

$$M_{kv\_cache} = 2 \times N_{layers} \times N_{kv} \times D_{head} \times L_{seq} \times B \times S_{prec}$$

- 系数 2 代表同时存储 Key 和 Value。
- 使用 GQA 时 $N_{kv} < N_{heads}$，直接减小了这一项。

# 五、MiniMind 注意力机制实现

下方代码实现了上述所有优化。为便于理解，每个关键步骤都标注了其对应的理论章节。

## 5.1 GQA 的 KV 头扩展（→ §2.3）

在 GQA 中，KV 头数少于 Q 头数，计算注意力前需要将 KV 头通过重复扩展到与 Q 相同的头数：

```python
def repeat_kv(x: torch.Tensor, n_rep: int) -> torch.Tensor:
    """
    GQA 核心操作：将少量 KV heads 重复扩展以匹配 Q heads 数量（→ §2.3 GQA）

    理论回顾：GQA 将 H 个 Q 头分成 G 组，每组共享一组 KV。
    计算注意力时，需要将 G 组 KV 各重复 H/G 次，使维度对齐。
    这不会增加 KV Cache 大小（缓存中仍只存 G 组），仅在计算时临时扩展。

    示例：8 个 Q heads / 2 个 KV heads → n_rep=4，每个 KV head 重复 4 次
    [B, L, 2, D] → [B, L, 2, 1, D] → [B, L, 2, 4, D] → [B, L, 8, D]
    """
    bs, slen, num_kv_heads, head_dim = x.shape
    if n_rep == 1:  # MHA 模式：KV 头数 = Q 头数，无需重复
        return x
    return (
        x[:, :, :, None, :]                                    # 插入重复维度
        .expand(-1, -1, -1, n_rep, -1)                         # 沿新维度复制
        .reshape(bs, slen, num_kv_heads * n_rep, head_dim)     # 合并为完整头数
    )
```

## 5.2 Attention 模块

```python
class Attention(nn.Module):
    """
    多头注意力模块，整合了三大优化：
    - GQA (§2.3)：减少 KV 头数，降低缓存显存
    - KV Cache (§1.2)：缓存历史 KV，避免重复计算
    - Flash Attention (§3)：IO 优化的注意力实现

    核心公式：Attention(Q, K, V) = softmax(Q·Kᵀ / √d_k) · V
    """

    def __init__(self, args: MiniMindConfig):
        super().__init__()

        # ======== GQA 配置（→ §2.3）========
        # 如果未指定 KV 头数，退化为 MHA（KV 头数 = Q 头数）
        self.num_key_value_heads = (
            args.num_key_value_heads
            if args.num_key_value_heads is not None
            else args.num_attention_heads       # 未指定时退化为 MHA（→ §2.1）
        )
        assert args.num_attention_heads % self.num_key_value_heads == 0

        self.n_local_heads = args.num_attention_heads             # Q 头数 = H
        self.n_local_kv_heads = self.num_key_value_heads          # KV 头数 = G（GQA 分组数）
        self.n_rep = self.n_local_heads // self.n_local_kv_heads  # 每组 KV 需重复次数 = H/G
        self.head_dim = args.hidden_size // args.num_attention_heads  # 单头维度 d_k

        # ======== Q/K/V/O 线性投影 ========
        # Q 投影：hidden_size → H * d_k（全量 Q 头）
        self.q_proj = nn.Linear(args.hidden_size, args.num_attention_heads * self.head_dim, bias=False)
        # K/V 投影：hidden_size → G * d_k（仅 G 组 KV 头，这就是 GQA 省显存的关键）
        self.k_proj = nn.Linear(args.hidden_size, self.num_key_value_heads * self.head_dim, bias=False)
        self.v_proj = nn.Linear(args.hidden_size, self.num_key_value_heads * self.head_dim, bias=False)
        # 输出投影：H * d_k → hidden_size
        self.o_proj = nn.Linear(args.num_attention_heads * self.head_dim, args.hidden_size, bias=False)

        # ======== Dropout ========
        self.attn_dropout = nn.Dropout(args.dropout)    # 注意力权重的 dropout
        self.resid_dropout = nn.Dropout(args.dropout)   # 残差连接的 dropout
        self.dropout = args.dropout                     # dropout 率

        # ======== Flash Attention 开关（→ §3）========
        # 需要 PyTorch 2.0+ 且配置中启用
        self.flash = (
            hasattr(torch.nn.functional, "scaled_dot_product_attention") and args.flash_attn
        )

    def forward(
        self,
        x: torch.Tensor,                                                         # [B, L, D]
        position_embeddings: tuple[torch.Tensor, torch.Tensor],                  # RoPE (cos, sin)
        past_key_value: Optional[Tuple[torch.Tensor, torch.Tensor]] = None,      # KV Cache
        use_cache: bool = False,
        attention_mask: Optional[torch.Tensor] = None,                           # [B, L]
    ):
        bsz, seq_len, _ = x.shape

        # ======== 步骤 1：Q/K/V 投影 ========
        # 将输入 x 投影到 Q/K/V 空间
        xq = self.q_proj(x)    # [B, L, H*d_k]  — 全量 Q
        xk = self.k_proj(x)    # [B, L, G*d_k]  — GQA: 仅 G 组 KV（→ §2.3）
        xv = self.v_proj(x)    # [B, L, G*d_k]

        # 重塑为多头格式
        xq = xq.view(bsz, seq_len, self.n_local_heads, self.head_dim)      # [B, L, H, d_k]
        xk = xk.view(bsz, seq_len, self.n_local_kv_heads, self.head_dim)   # [B, L, G, d_k]
        xv = xv.view(bsz, seq_len, self.n_local_kv_heads, self.head_dim)   # [B, L, G, d_k]

        # ======== 步骤 2：应用 RoPE 位置编码 ========
        cos, sin = position_embeddings
        xq, xk = apply_rotary_pos_emb(xq, xk, cos, sin)

        # ======== 步骤 3：KV Cache 处理（→ §1.2）========
        # Decode 阶段：将当前 Token 的 KV 拼接到历史缓存上
        if past_key_value is not None:
        	# 在（时间）序列维度（dim=1）上拼接 [B, past_L+L, G, d_k]
            xk = torch.cat([past_key_value[0], xk], dim=1)
            xv = torch.cat([past_key_value[1], xv], dim=1)
        # 保存新的 KV 缓存供下一步使用
        past_kv = (xk, xv) if use_cache else None

        # ======== 步骤 4：GQA KV 头扩展（→ §2.3）========
        # 将 G 组 KV 重复扩展到 H 组，与 Q 头数对齐
        # 注意：这只是计算时的临时扩展，KV Cache 中仍只存 G 组
        xq = xq.transpose(1, 2)                              # [B, H, L, d_k]
        xk = repeat_kv(xk, self.n_rep).transpose(1, 2)       # [B, H, kv_len, d_k]
        xv = repeat_kv(xv, self.n_rep).transpose(1, 2)       # [B, H, kv_len, d_k]

        # ======== 步骤 5：计算注意力 ========
        if self.flash and seq_len > 1 and past_key_value is None and \
           (attention_mask is None or torch.all(attention_mask == 1)):
            # --- Flash Attention 路径（→ §3）---
            # 条件：训练/Prefill 阶段（seq_len > 1），无 KV Cache，无复杂掩码
            # 分块在 SRAM 中计算，避免将 L×L 注意力矩阵写入 HBM
            output = F.scaled_dot_product_attention(
                xq, xk, xv,
                attn_mask=None,
                dropout_p=self.dropout,
                is_causal=True,             # 自动应用因果掩码
            )
        else:
            # --- 标准注意力路径 ---
            # 5a. 计算注意力分数：Q·Kᵀ / √d_k
            scores = (xq @ xk.transpose(-2, -1)) / math.sqrt(self.head_dim)

            # 5b. 因果掩码：上三角置为 -∞，防止看到未来信息（→ §1.1 因果性）
            scores[:, :, :, -seq_len:] += torch.triu(
                torch.full((seq_len, seq_len), float("-inf"), device=scores.device),
                diagonal=1,
            )

            # 5c. Padding 掩码（如有）
            if attention_mask is not None:
                mask = attention_mask.unsqueeze(1).unsqueeze(2)   # [B, 1, 1, L]
                scores = scores + (1.0 - mask) * -1e9             # 0 位置 → -∞

            # 5d. Softmax → Dropout → 加权求和
            scores = F.softmax(scores.float(), dim=-1).type_as(xq)
            scores = self.attn_dropout(scores)
            output = scores @ xv                                  # [B, H, L, d_k]

        # ======== 步骤 6：合并多头 & 输出投影 ========
        output = output.transpose(1, 2).reshape(bsz, seq_len, -1)   # [B, L, H*d_k]
        output = self.resid_dropout(self.o_proj(output))            # → [B, L, D]

        return output, past_kv
```

> [!info] 代码与理论的对应总结
>
> | 代码位置 | 对应理论 | 关键变量 |
> |---------|---------|---------|
> | `__init__` 中 KV 头数配置 | §2.3 GQA | `n_local_kv_heads`（G 组）vs `n_local_heads`（H 个 Q 头） |
> | K/V 投影维度 `G*d_k` | §2.3 GQA 省显存 | `k_proj` , `v_proj` 输出维度小于 `q_proj` |
> | `repeat_kv()` | §2.3 计算时临时扩展 | 缓存仍只存 G 组，计算时扩展到 H 组 |
> | `past_key_value` 拼接 | §1.2 KV Cache Decode 阶段 | `torch.cat` 在 seq 维度追加 |
> | `use_cache` 返回 `past_kv` | §1.2 KV Cache 机制 | 每步返回最新 KV 供下一步复用 |
> | `F.scaled_dot_product_attention` | §3 Flash Attention | 仅在训练/Prefill 时启用 |
> | 标准路径的 `triu` 掩码 | §1.1 因果性 | 上三角 → $-\infty$，确保不看未来 |
> | §4 显存公式中的 $N_{kv}$ | §2.3 GQA | GQA 时 $N_{kv} = G < H$，直接减小 KV Cache |
